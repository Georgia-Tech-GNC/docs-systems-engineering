# Writing Style

Documentation in this repository must be concise, specific, and useful to an engineering reader.

## Principles

- Be brutally efficient. Delete fluff. Explain only what the reader needs to act correctly.
- Define a term by what it is, rather than by what it is not. Use counterexamples only after the definition, and only when they prevent a likely misunderstanding.
- Avoid buzzwords and generic process language.
- Keep lists short. If a sentence needs more than three listed items, split the idea into smaller statements.
- Preserve flow. A reader should not need to jump ahead to understand the current section.
- Give each document one clear purpose. Separate documents must not overlap in purpose.

## Definitions

Artifact and process definitions should answer three questions:

- What is this?
- What is it for?
- When is it required?

Add more detail only when it changes how the artifact is written, reviewed, approved, or linked.

## Structure

Put background before rules only when the background is needed to understand the rule. Otherwise, state the rule first.

Use headings to separate distinct questions the reader is trying to answer. If two sections answer the same question, merge them.

## Links And References

Link to the authoritative definition instead of restating it. Restatement creates drift.

Do not send the reader forward for basic context. A forward link is acceptable for deeper detail, but the current section must still make sense on its own.

## Tone

Use plain engineering language. Prefer direct verbs and concrete nouns.

Avoid motivational language, filler transitions, and vague claims of importance. The documentation should earn attention by being clear.
